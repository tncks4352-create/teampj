// ============================================
// 🎯 3. 주요 설계 결정사항
// ============================================

1️⃣ 화면 관리 구조
   결정: ScreenManager를 중앙 집중식으로 통합
   이유: 화면 전환 로직 단순화, 상태 추적 용이
   장점: 모든 화면 전환이 일관되게 처리됨

2️⃣ 모듈 간 통신
   결정: EventSystem (pub/sub 패턴)
   이유: 모듈 간 느슨한 결합, 독립성 유지
   장점: UI 모듈들이 독립적으로 구현 가능

3️⃣ 신력 시스템
   결정: 별도 시스템 (골드와 분리)
   이유: 전투 밸런스, 확장성, 명확한 구분
   장점: 신력 회복 속도 조절 가능

4️⃣ 상태 중앙화
   결정: 모든 게임 상태를 gameState 단일 객체로 관리
   이유: 데이터 동기화 단순화, 저장/로드 용이
   장점: 프로젝트 복잡도 관리

5️⃣ 기존 코드 호환성
   결정: 전투 시스템은 최대한 유지, 참조만 변경
   이유: 기존 파일럿 기능 보존, 새 기능과의 조화
   장점: 롤백 위험 최소화

// 🔄 5. 플레이 흐름 변경

기존 (픽셀 디펜스):
게임 시작 → [타이틀] → [로비] → 스테이지 선택 → [전투]

신규 (신화창조):
게임 시작 → [메인 로비] ──┬→ [신 모집]
                      ├→ [성장]
                      ├→ [상점]
                      ├→ [미션]
                      └→ [전투 시작] → [덱 편성] → [전투] → [보상]

메인 로비 특징:
- 선택 신을 크게 중앙 표시 (Lv, 별급, 진화 게이지)
- 상단: 골드/다이아
- 중앙: 신 카드 + 전투 시작 버튼
- 하단: 4개 탭 메뉴
